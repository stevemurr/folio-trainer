# Spec: Daily Predictive Asset Allocation Pipeline (v1)

## Purpose
Build a point-in-time daily pipeline that:
1. prepares training data for a predictive model that learns the next target asset allocation across a fixed ticker universe, and
2. trains/evaluates a model that predicts the target allocation for risk-adjusted return.

This spec is written so an implementation agent can start immediately with sensible defaults and minimal follow-up.

---

## Default assumptions the agent should use unless overridden
- **Universe**: fixed user-supplied ticker list plus a synthetic `CASH` sleeve.
- **Constraints**: long-only, fully invested, no leverage, no shorting.
- **Weight simplex**: all weights >= 0 and sum(weights) = 1.
- **Signal timing**: features are frozen after market close on date `t` using an as-of cutoff at **18:00 America/New_York**; model predicts target weights for execution on the next trading session `t+1`.
- **Primary label horizon**: 20 trading days.
- **Additional horizons**: 5 and 60 trading days (optional multi-head labels).
- **Teacher labels**: generated by candidate portfolio search using a mixture of Dirichlet samples + deterministic portfolios.
- **Execution policy in evaluation**: daily signal generation, but trade only if target change exceeds a minimum band; rebalance partially toward target.
- **Baselines required**: equal weight, inverse volatility / risk parity approximation, min-vol approximation, and previous-weights hold.
- **Public exogenous data**: SEC EDGAR fundamentals/filings, FRED/ALFRED vintaged macro series, Treasury daily yield curve, Kenneth French daily factors, FRED VIX.
- **Price history for the universe**: use the user’s existing adjusted OHLCV source or a separately approved market-data provider. Do not scrape consumer websites.

---

## Decisions to confirm with the user
If the user does not answer, proceed with the defaults above.
1. Exact ticker universe.
2. Execution price convention: next open, next close, or VWAP proxy.
3. Max single-name weight cap (default 25%).
4. Whether to include `CASH` as literal 0-return cash or 3-month Treasury proxy (default: 3-month Treasury proxy).
5. Broker cost assumptions if the user already has them; otherwise use configurable heuristics.

---

## Non-goals for v1
- No leverage.
- No shorting.
- No options/futures.
- No intraday execution optimization.
- No LLM-based filing text embeddings in v1.
- No dynamic universe expansion/contraction in v1.

# Part 1 — Training data preparation

## 1.1 Deliverable
Produce a reproducible point-in-time dataset with:
- raw source tables,
- feature tables,
- teacher candidate tables,
- distilled target-label tables,
- split metadata,
- data quality reports.

The output must be sufficient to train at least one direct weight-prediction model and one optional utility-scoring model.

---

## 1.2 Point-in-time rules (must follow strictly)
1. **No lookahead**. For as-of date `t`, every feature must have a source timestamp `<= cutoff_ts(t)`.
2. **Daily market features** may use same-day close data known by the cutoff.
3. **Macro data with revisions** must be sourced from ALFRED/vintage-aware retrieval and joined using the latest release available as of `cutoff_ts(t)`.
4. **SEC filing/fundamental data** must be joined by public filing acceptance time / public availability, never by fiscal period end alone.
5. **Training/validation/test boundaries** must be purged by at least `max_feature_lookback + max_label_horizon` trading days.
6. **All transforms fit only on train** (scalers, imputers, PCA, etc.).

---

## 1.3 Canonical time/index definitions
- `trade_date`: exchange trading day.
- `cutoff_ts`: `trade_date` at 18:00 America/New_York.
- `prediction_date`: next exchange trading day after `trade_date`.
- `execution_date`: same as `prediction_date` for v1.
- `horizon_h`: forward window length in trading days, default in `{5,20,60}`.

Use NYSE/XNYS trading calendar semantics unless the universe requires a different exchange calendar.

## 1.7 Feature engineering spec

### 1.7.1 Asset-level price/volume features
For each ticker on each `asof_date`, compute at minimum:
- returns: `ret_1, ret_5, ret_20, ret_60, ret_120`
- realized volatility: `vol_5, vol_20, vol_60, vol_120`
- downside volatility: `downvol_20, downvol_60`
- drawdown metrics: `dd_20, dd_60, dd_120`
- momentum / reversal: `mom_20, mom_60, mom_120, rev_5`
- rolling beta to universe equal-weight index: `beta_eqw_60, beta_eqw_120`
- rolling correlation to universe equal-weight index: `corr_eqw_20, corr_eqw_60`
- idiosyncratic volatility from rolling market regression: `idio_vol_60`
- liquidity: `dollar_vol_20`, `dollar_vol_60`, `amihud_20`
- volume shock features: z-scores of volume and dollar volume
- current portfolio context: `prev_live_weight`, `prev_target_weight`, `delta_from_prev_target`

### 1.7.2 Cross-asset structure features
For each `asof_date`, compute from the whole universe return matrix:
- equal-weight universe return
- covariance summaries for 20/60-day windows
- average pairwise correlation for 20/60-day windows
- first 3 PCA explained variance ratios for the cross-section
- per-ticker first 3 PCA loadings for current window
- concentration of recent winners/losers (top-decile breadth if universe size allows)
- risk parity denominator inputs: rolling vol and covariance summaries

### 1.7.3 Market/regime features
At minimum:
- VIX level and 5/20-day changes
- Treasury yields: 3m, 2y, 10y, and slope terms `10y-2y`, `10y-3m`
- Fama-French daily factors: `Mkt-RF, SMB, HML`
- daily momentum factor `Mom`
- optional macro series via ALFRED (forward-filled from release availability): CPI, unemployment, industrial production, fed funds

### 1.7.4 Fundamentals/event features
Joined point-in-time from SEC data where available:
- days since last 10-Q / 10-K
- filing count in last 365 days
- basic valuation proxies if derivable from existing market cap + company facts
- profitability / leverage proxies if company facts are available and quality is acceptable
- missingness flags for every fundamental family

### 1.7.5 Missing-data policy
- Forward-fill only where economically valid and point-in-time safe.
- Then impute remaining missing values with train-period median.
- Add missingness indicator columns for every imputed feature group.
- Reject any training day where fewer than 80% of required universe members have core price features.

## 1.8 Teacher label generation

### 1.8.1 Goal
For each `asof_date t` and horizon `H`, generate candidate allocations and score them using realized forward performance, transaction costs, turnover, and concentration penalties.

### 1.8.2 Portfolio constraints
Default constraints:
- long-only
- sum(weights) = 1
- `weight_i <= weight_cap_i`
- `weight_cash >= 0`
- no leverage
- no shorting

### 1.8.3 Candidate generation
For each `asof_date` and horizon `H`, build candidates from:
1. deterministic portfolios:
   - previous live weights
   - previous target weights
   - equal weight
   - cash only
   - inverse volatility / risk parity approximation
   - minimum variance approximation
2. random portfolios sampled from a **mixture of Dirichlet distributions**
3. local perturbations around the previous live weights and the best deterministic seeds

Default candidate counts:
- deterministic seeds: all listed above
- Dirichlet random candidates: 5,000 per day per horizon
- alpha mixture: `[0.05, 0.20, 1.0, 5.0]`
- local perturbations: 100 candidates around each of the top 5 deterministic seeds

### 1.8.4 Forward scoring function
Let `w` be a candidate target portfolio on day `t`.

Definitions:
- `r_{t+1:t+H}` = realized forward daily asset returns over horizon `H`
- `r_p` = daily portfolio return series for `w`
- `rf` = daily cash/risk-free proxy return
- `prev_live_w_t` = portfolio weights held immediately before rebalancing on `t+1`
- `turnover(w)` = `0.5 * sum(abs(w - prev_live_w_t))`
- `HHI(w)` = `sum(w_i^2)`

Default teacher objective:

```text
score(w) = sharpe_H(r_p - rf)
           - lambda_turnover * turnover(w)
           - lambda_cost * est_cost(w)
           - lambda_concentration * HHI(w)
```

Where:
- `sharpe_H` = mean daily excess return over the horizon divided by std dev + epsilon.
- `est_cost(w)` is a configurable trading-cost heuristic.

### 1.8.5 Trading-cost heuristic for teacher labels
Implement a configurable cost function:

```text
est_cost_bps(w) = commission_bps
                + regulatory_bps
                + spread_bps_proxy
                + impact_coeff * sqrt( trade_notional / max(ADV20, adv_floor) )
```

Where:
- `trade_notional` is derived from target change vs previous live weights.
- `ADV20` is 20-day average dollar volume for the asset.
- If quote spread is unavailable, `spread_bps_proxy` should come from a liquidity bucket based on ADV and price level.

The exact constants must live in config, not in code.

### 1.8.6 Sequential dependency / live-weights simulation
Teacher labels must be generated sequentially through time:
1. initialize `prev_live_weights` on the first trainable date (default: equal weight including cash sleeve set to 0 unless user prefers a different start);
2. optimize candidate score for day `t` using the current `prev_live_weights`;
3. store the best candidate as `hard_target`;
4. update `prev_live_weights` for day `t+1` by applying the chosen trade rule and market drift.

This makes turnover and cost penalties path-dependent in a realistic way.

### 1.8.7 Soft-target distillation
Keep top `K` candidates per day/horizon (default `K=20`).
Create soft labels:

```text
p_k = softmax( objective_k / temperature )
soft_target = sum_k p_k * w_k
confidence = best_objective - objective_at_rank_K
```

Store:
- `hard_target_weights`
- `soft_target_weights`
- `teacher_confidence`
- all candidate rows for optional utility-model training

## 1.9 Splits and leakage control

### 1.9.1 Static split (required)
- `train`: first 70% of dates
- `val`: next 15% of dates
- `test`: final 15% of dates

Apply purge buffer:

```text
purge_days = max(max_feature_lookback, max_label_horizon)
```

With current defaults, that is at least 120 trading days.

### 1.9.2 Walk-forward splits (required)
Also produce rolling folds such as:
- train 5 years / validate 1 year / test 1 year, rolling forward by 1 year

The evaluator must report both single holdout and walk-forward results.

## 2.2 Required baselines
Implement and evaluate these baselines before any learned model:
1. equal weight
2. previous-live-weights hold
3. inverse volatility / risk parity approximation
4. minimum variance approximation using rolling covariance
5. teacher hard-label replay (oracle upper bound, cost-aware but not deployable)

---

## 2.3 Required v1 model: direct weight prediction

### 2.3.1 Model form
Implement a cross-sectional scoring model that predicts one raw score per asset and converts scores to weights via masked softmax.

For each `asof_date` and asset row:
- input = asset features + shared market features + current portfolio context
- model outputs raw score `s_i`
- convert to weights:

```text
w_i = mask_i * exp(s_i / tau) / sum_j mask_j * exp(s_j / tau)
```

Where:
- `mask_i` = 0 if asset unavailable on that date, else 1
- `tau` is a configurable temperature parameter

Represent `CASH` as a normal asset row with its own features.

### 2.3.2 Recommended architectures
Implement both of these if time allows; at minimum implement the first.
1. **Gradient-boosted trees per asset score** (e.g., LightGBM/XGBoost style rank/regression setup) feeding a softmax normalization.
2. **Tabular neural network** with shared per-asset encoder + pooling/shared market context + per-asset head.

For v1 selection, choose the model with the best validation net Sharpe after backtest simulation, not merely the lowest training loss.

### 2.3.3 Loss function
Primary loss:
- KL divergence or cross-entropy from predicted weights to `soft_target_weights`

Auxiliary loss terms:
- L2 penalty on day-to-day weight changes
- concentration penalty if needed
- optional hard-target MSE / MAE term

Sample weighting:
- weight each training example by `teacher_confidence` clipped to a reasonable range

### 2.3.4 Multi-horizon option
If implementing multi-head training:
- shared backbone
- one output head per horizon in `{5,20,60}`
- production prediction can default to the 20-day head or a configurable blend

---

## 2.4 Optional v1b model: utility scorer
This is optional but strongly recommended if time allows.

Train a second model on:

```text
(state_features, candidate_weights) -> teacher_objective_total
```

Use it to:
- score candidate portfolios at inference time
- compare predicted objective ranking vs teacher ranking
- provide a hedge against noisy single-label distillation

Required if implemented:
- pairwise ranking metric (Spearman / Kendall on candidate ranks)
- top-k recall of teacher-best candidate

## 2.5 Training procedure

### 2.5.1 Preprocessing
- fit scalers/imputers on train only
- persist preprocessing artifacts with the model
- ensure identical column order between train/inference

### 2.5.2 Hyperparameter search
Required search dimensions for the direct model:
- learning rate
- max depth / hidden width
- regularization strength
- softmax temperature
- turnover regularization weight
- label horizon or multi-head blend

Use time-series-safe validation only.

### 2.5.3 Early stopping and model selection
Select the best checkpoint by:
1. validation backtest net Sharpe,
2. then validation max drawdown,
3. then turnover,
4. then pure prediction loss.

### 2.5.4 Reproducibility
- set and log all random seeds
- log package versions
- save training config with each run
- save feature manifest / schema hash

## 2.6 Inference and execution policy
The model produces a target weight vector each day.

### 2.6.1 Default trade rule for backtests
Given `pred_target` and `prev_live_weights`:
1. compute `l1_change = 0.5 * sum(abs(pred_target - prev_live_weights))`
2. if `l1_change < rebalance_band`, do not trade
3. else execute:

```text
exec_target = prev_live_weights + partial_rebalance_alpha * (pred_target - prev_live_weights)
```

Default config values:
- `rebalance_band = 0.02`
- `partial_rebalance_alpha = 0.50`

These must be configurable and reported.

### 2.6.2 Post-trade drift
After execution, live weights drift according to realized next-period returns until the next rebalance decision.
